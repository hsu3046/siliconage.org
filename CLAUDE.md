# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project Overview

The Silicon Age ‚Äî interactive knowledge graph of 100 years of digital history. Single-page React app deployed as a static site on Vercel. ~200 entities (Companies, Persons, Technologies) connected by typed links, visualized as a D3 force graph + chronological timeline + filterable card grid + relationship explorer.

## Commands

```bash
npm run dev               # Vite dev server on http://localhost:3000
npm run build             # Production build (vite build)
npm run preview           # Preview the production build

# AI cache generation (requires VITE_GOOGLE_API_KEY in .env)
npm run generate-cache    # Calls Gemini for every node, writes services/staticCache.ts

# i18n pipeline ‚Äî extract source strings from constants.ts, then translate via AI
npm run i18n:sync-all     # extract + translate nodes, links, events (all three locales)
npm run i18n:sync         # nodes only
npm run i18n:sync-links   # links only
npm run i18n:sync-events  # events only
```

No test suite, no lint script. TypeScript is strict ‚Äî rely on `tsc` errors surfaced through `vite build`.

Debug dashboard: `http://localhost:3000/debug` (dev-only, lazy-loaded so it's excluded from the production bundle).

## Architecture

### Data lives in one file
`constants.ts` (~155 KB) is the source of truth for the entire graph: every node, link, era, and event is a hand-curated entry. There is no database and no API. Adding or editing an entity = editing `constants.ts` and re-running the i18n sync.

`types.ts` defines the schema. Key shapes:
- `NodeData`: `{ id, label, category (COMPANY/PERSON/TECHNOLOGY), year, impactRole, techCategoryL1/L2, ... }`
- `LinkData`: `{ source, target, type (POWERS/CREATES/CONTRIBUTES/ENGAGES), arrow, icon, strength, story?, startYear?, endYear? }`
- Roles are strongly typed enums per category (`PersonRole`, `TechRole`, `CompanyCategory`).

Authoring rules for nodes and links: see `docs/NodeLinkRules.md` and `docs/LabelingRules.md`. Follow these when editing `constants.ts` ‚Äî IDs, role assignments, and link types follow specific conventions.

### Computed ranking
`utils/ranking.ts ‚Üí calculateSiliconRank()` is run **once at module load** in `App.tsx` against `INITIAL_DATA`. It mutates each node with `_radius` / `_zoneRadius` derived from link weights √ó category multipliers √ó tech-role bonuses. Don't run this per-render ‚Äî graph sizing must stay stable across navigations.

### Four views over the same dataset
`App.tsx` holds top-level state (`viewMode`, `selectedNode`, `focusNodeId`, locale, URL sync) and renders one of:
- `MapView` ‚Äî D3 force-directed graph (the main view)
- `HistoryView` ‚Äî chronological timeline
- `CardView` ‚Äî filterable card grid
- `LinksView` ‚Äî pairwise relationship explorer

URL state is round-tripped via React Router 7 (`hooks/useUrlState.ts`) ‚Äî deep links to `/node/:id` and view-mode routes must keep working. Initial-load sync uses `isInitialLoadRef` to avoid clobbering router state before the URL is read.

### AI deep dives (BYOK)
`services/geminiService.ts` powers on-demand entity analysis via Google Gemini. Two layers:
1. **Static cache** (`services/staticCache.ts`) ‚Äî pre-generated by `scripts/generateCache.ts`. The app works fully offline-AI with this file alone.
2. **Live API** ‚Äî only used when the user has pasted their own key into About ‚Üí API Key Settings. Key is stored in `localStorage` (`silicon_age_api_key`), never sent to any server we control.

When adding new nodes, either regenerate the static cache or accept that those nodes fall through to "live key required."

### i18n architecture
Three locales: `en`, `ko`, `ja`. Translations split into four JSON files per locale under `locales/<lang>/`:
- `ui.json` ‚Äî UI strings (hand-edited)
- `nodes.json`, `links.json`, `events.json` ‚Äî generated by `scripts/translate-*.ts` from `constants.ts`

Runtime: `utils/i18n.ts` + `hooks/useLocale.ts`. Locale changes are async (`loadLocale`) ‚Äî components must handle the pre-loaded state. `translateNode` / `translateLink` / `translateEvent` look up by ID and fall back to the original (English-source) string.

**When you edit a node label, description, link story, or event in `constants.ts`, run `npm run i18n:sync-all`** to regenerate translations. The extract scripts diff against the existing locale files so only changed entries hit the translation API.

### Path alias
`@/*` resolves to repo root (see `vite.config.ts` and `tsconfig.json`). Use `@/components/...` over deep relative paths.

## Conventions

- **Edit `constants.ts`, not generated locale files.** Locale JSON is rebuilt by the i18n pipeline.
- **No backend.** Everything ships as static assets. Don't introduce server routes or API handlers ‚Äî this repo is not Next.js.
- **Strict TS, no `any`.** When extending `NodeData` / `LinkData`, update `types.ts` first.
